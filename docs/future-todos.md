## Self-update download paths (cross-script)

These bullets cover correctness and resilience gaps in the `download_script` family of helpers and their callers across the repo. The 2026-05-05 backport (branch `backport-self-update-fixes`) closed 10 of the gaps tracked in earlier revisions of this file (commits `8746154` through `a557927`). The items below remain deferred for the reasons noted.

- **Tighten shebang validation from `head -n 10 | grep -q "^#!/"` to first-line + CRLF rejection.** The current loose validation passes if ANY line in the first 10 starts with `#!/` — meaning an HTML 4xx/maintenance page that happens to mention `#!/bin/sh` in a code snippet (lines 2–10) qualifies as a "valid script" and gets `chmod +x`'d and `exec`'d. Combined with the `-fsSL` HTTP-error bug (now fixed by the 2026-05-05 backport — `-fsSL` → `-sSL`), an attacker-controlled or upstream-misconfigured response can become RCE. Separately, a CRLF-terminated shebang (`#!/usr/bin/env bash\r\n`) passes the prefix check but `exec` then fails with `bash\r: not found`, leaving the script replaced on disk but unrunnable. Replace with bash `read` of first 2 lines and explicit `[[ "$first_line" == "#!"* ]]` check, plus `[[ "$first_line" == *$'\r' ]]` rejection. Affected sites (10+ matches): `kubernetes/utils-k8s.sh` (2), `github/gh_org_delete_issues.sh` (2), `github/gh_org_delete_repos.sh` (2), `github/gh_org_copy.sh` (2), `lxc/utils-lxc.sh` (2), plus the `download_script` definitions in `system-setup/utils-sys.sh`, `lxc/_download-lxc-scripts.sh`, `llm/utils-llm.sh`, `llm/_download-ollama-scripts.sh`. The sister `system-setup-private` repo's `utils/tools-update.sh:download_file` already implements this — copy that pattern.

- **Map wget exit codes to actionable error messages instead of collapsing all failures.** Most `download_script` wget branches in this repo use `wget -q ... 2>/dev/null` and treat any non-zero exit as a generic "Download failed (network/timeout)". `wget(1)` documented exit codes do distinguish: 4=network, 5=SSL, 6=auth, 8=server (HTTP 4xx/5xx), so a user with an expired credential on a wget-only host gets misdirected to look at their network. Capture the exit code via `wget … || wget_exit=$?` then `case` on it: 4|5 → "Network/SSL error", 6 → "Authentication failed — check token", 8 → "Server error — check token and repo access", * → "Download failed (wget exit $wget_exit)". The 2026-05-05 backport already captures `wget_exit` but only renders it generically — adding the `case` arms is mechanical. Affected sites: every helper library and standalone with a `download_script` body.

## Self-update / module-update diff display (cross-script)

- **Sanitize non-SGR ANSI escape sequences from diff output to defend against terminal-injection.** When a self-update fetches attacker-controllable file content (the remote script could be hostile under a repo-write compromise) and renders it through `diff --color=always | less -RFX` (the pattern landed in the 2026-05-05 backport), cursor-movement escapes embedded in the file content render directly to the user's terminal. Threats: (1) cursor-up + erase-line sequences can repaint over the diff context the user just scrolled past, hiding malicious lines; (2) more concretely, after the diff ends but before the y/N prompt prints, the file content can emit cursor escapes that draw a fake "no changes detected" line OVER the real overwrite-and-restart prompt, leveraging default-yes prompts to one-keypress acceptance. `less -R` deliberately passes ANSI through and does NOT filter cursor-movement codes; only SGR (color) is safe. The private `system-setup-private` repo's review (2026-04-27) shipped a `_sanitize_ansi` sed helper that strips two patterns: CSI sequences whose final byte is NOT `m` (cursor moves, mode toggles, erase line) and BEL-terminated OSC sequences (set-title, etc.), keeping SGR codes intact. Apply at every `show_diff_box` callsite. Pattern: pipe diff output through `sed -E -e "s/${ESC}\\[[0-9;?]*[^0-9;?m]//g" -e "s/${ESC}\\][^${BEL}]*${BEL}//g"` where `ESC=$'\033'` and `BEL=$'\007'` (literal bytes from bash so the regex stays portable between GNU and BSD sed).

## Config-rewrite sites (atomic-rename gap)

- **Atomic-rename for config-rewrite sites under elevation.** `normalize_trailing_newlines()` in both `utils-sys.sh` and `utils-k8s.sh`, plus `update_config_line()` in `utils-k8s.sh`, currently use bare `mktemp` (`/tmp` tmpfs) then cross-FS `mv` to `$file` (often `/etc/...` via `run_elevated`). A SIGKILL or power-loss mid-`mv` can leave a truncated config file. Atomic same-FS rename requires either (a) `mktemp` adjacent to `$file` AND running `mktemp` under the same elevation as the eventual `mv` (complicates the function's calling convention), or (b) writing through a privileged helper that handles both the temp creation and rename atomically. Out of scope for the 2026-05-05 self-update backport; Pattern E3 (bookkeeping-only inline `mktemp` + `TEMP_FILES+=()`) preserved current cross-FS behavior at those sites. Surfaced as a deliberate scope decision in plan `Reviewed: Backport 10 self-update / temp-file / prompt-UX fixes`.

- **`utils-sys.sh:update_config_line` Pattern E3 gap.** During the 2026-05-05 backport, `utils-k8s.sh:update_config_line` was converted to Pattern E3 (bare `mktemp` + `TEMP_FILES+=()` inline so the EXIT trap can reap a SIGINT-leaked temp). The analogous `utils-sys.sh:update_config_line` (around `system-setup/utils-sys.sh:885`) was NOT converted — the plan only listed the utils-k8s site. Result: two libraries with different tracking behavior for the same function. Low impact (config-rewrite to `/etc/...` via `run_elevated` is cross-FS anyway, the temp goes to `$TMPDIR` tmpfs and is reaped at boot), but inconsistent with the sibling library. Surfaced by Stage 2 code review on branch `backport-self-update-fixes`. Future fix: apply Pattern E3 to `utils-sys.sh:update_config_line` to align the two libraries.
