## system-setup/utils-sys.sh

- **Add an HTTP timeout to `download_script` and fix the `-fsSL || echo "000"` HTTP-error bug.** `download_script` calls curl/wget with no `--max-time` (curl) or `--timeout=` (wget). On a flaky network or DNS sinkhole, callers can hang for the system default (5+ minutes) before falling back. Separately, the curl branch's `-fsSL ... -w "%{http_code}" ... || echo "000"` produces `"404000"` (not `"404"`) on HTTP errors because curl with `-f` still writes `%{http_code}` via `-w` AND exits non-zero, so the `|| echo "000"` appends — breaking the case match. The private `system-setup-private` repo's pings-loop.sh self-update plan (2026-04-27) shipped both fixes atomically across `utils/pings-loop.sh`, `utils/tools-update.sh`, and `tmux/utils-tmux.sh` — backport to the public repo for parity. Add `--max-time 15` (curl) and `--timeout=15` (wget) — a single 15s knob covers DNS + connect + read; drop `-f` from the curl flags and only set `"000"` when the captured `http_status` is empty (signals a transport failure with no HTTP response). Verified via [everything.curl.dev/usingcurl/timeouts.html](https://everything.curl.dev/usingcurl/timeouts.html).

## Self-update download paths (cross-script)

These bullets cover correctness and resilience gaps in the `download_script` family of helpers and their callers across the repo. The private `system-setup-private` repo addressed each as part of its 2026-04-27 self-update review pass; backporting here keeps the two repos aligned and closes RCE-adjacent and reliability gaps.

- **Tighten shebang validation from `head -n 10 | grep -q "^#!/"` to first-line + CRLF rejection.** The current loose validation passes if ANY line in the first 10 starts with `#!/` — meaning an HTML 4xx/maintenance page that happens to mention `#!/bin/sh` in a code snippet (lines 2–10) qualifies as a "valid script" and gets `chmod +x`'d and `exec`'d. Combined with the `-fsSL` HTTP-error bug (already tracked above), an attacker-controlled or upstream-misconfigured response can become RCE. Separately, a CRLF-terminated shebang (`#!/usr/bin/env bash\r\n`) passes the prefix check but `exec` then fails with `bash\r: not found`, leaving the script replaced on disk but unrunnable. Replace with bash `read` of first 2 lines and explicit `[[ "$first_line" == "#!"* ]]` check, plus `[[ "$first_line" == *$'\r' ]]` rejection. Affected sites (10+ matches grepped 2026-04-27): `kubernetes/utils-k8s.sh` (2), `github/gh_org_delete_issues.sh` (2), `github/gh_org_delete_repos.sh` (2), `github/gh_org_copy.sh` (2), `lxc/utils-lxc.sh` (2), plus the `download_script` definitions in `system-setup/utils-sys.sh`, `lxc/_download-lxc-scripts.sh`, `llm/utils-llm.sh`, `llm/_download-ollama-scripts.sh`. Coordinate with the timeout + `-fsSL` backport in the `## system-setup/utils-sys.sh` entry above — single sweep covers both.

- **Wrap `mv -f temp_file local_script` in an explicit `if !` failure handler.** Bare `mv -f` under `set -e` aborts the script with no user-facing error on failure (read-only mount, full disk, permission denied), and the temp file is leaked unless a separate `TEMP_FILES`+EXIT trap reaps it. Replace with `if ! mv -f "$tmp" "$dest"; then print_error "✖ Failed to install update — keeping local version"; return 1; fi`. Affected sites (8 grepped 2026-04-27): `kubernetes/utils-k8s.sh` (2), `kubernetes/kubernetes-setup.sh` (1), `github/gh_org_copy.sh` (1), `github/gh_org_delete_issues.sh` (1), `github/gh_org_delete_repos.sh` (1), `lxc/_download-lxc-scripts.sh` (1), `lxc/utils-lxc.sh` (2). The private repo's `utils/tools-update.sh:self_update` and `tmux/utils-tmux.sh:check_for_updates` ship the same pattern.

- **Use `mktemp "${dest}.XXXXXX"` so `mv -f` is an atomic same-FS rename.** Bare `mktemp` (no template) lands the file in `$TMPDIR` (typically `/tmp` tmpfs); a subsequent `mv` to `$SCRIPT_DIR` (typically disk-backed) is cross-filesystem `copy + unlink`, which is NOT atomic. SIGKILL or power-loss mid-copy can leave a truncated script at the destination that the next caller will source/exec. Switching the mktemp template to be adjacent to the destination forces same-filesystem placement, so `mv` becomes a single `rename(2)` syscall — atomic. Affected: every site that uses `mktemp` then `mv -f` to a known destination. The bare-mktemp grepped sites (2026-04-27): `lxc/utils-lxc.sh` (2), `llm/utils-llm.sh` (2), `lxc/_download-lxc-scripts.sh` (1), `github/gh_org_*` (3), `llm/_download-ollama-scripts.sh` (1), `system-setup/utils-sys.sh:669`, `kubernetes/utils-k8s.sh` (3). Note that `system-setup/utils-sys.sh:820` (`local temp_file=$(mktemp)` for permissions backup) is a different use case where atomicity is irrelevant — only fix the self-update / module-update sites. The `utils/tools-update.sh:sync_claude_md_template` precedent in the private repo already does this for CLAUDE.md sync.

- **Track `mktemp` allocations via a `TEMP_FILES` array + `trap cleanup EXIT` for SIGINT/SIGTERM safety.** Several scripts use raw `mktemp` and rely on inline `rm -f` calls in each return path. SIGINT/SIGTERM during the window between `mktemp` and the matching `rm -f` leaks the temp. The repo already has the canonical pattern in `kubernetes/utils-k8s.sh`, `system-setup/utils-sys.sh`, and `utils/services-check.sh` (`TEMP_FILES=()` array + `make_temp_file` helper + EXIT trap). Backport the pattern to scripts that don't yet track temps: `lxc/utils-lxc.sh`, `llm/utils-llm.sh`, `lxc/_download-lxc-scripts.sh`, `llm/_download-ollama-scripts.sh`, `github/gh_org_delete_repos.sh`, `github/gh_org_copy.sh`, `github/gh_org_delete_issues.sh`. Note that bash does NOT fire EXIT on successful `exec`, so the happy update path (mv consumes temp, then exec) is unaffected — the trap is the safety net for interrupted runs. **UPDATE 2026-05-04:** The `make_temp_file` helper itself has a subshell-capture leak — every `tmp=$(make_temp_file …)` call site mutates the subshell's `TEMP_FILES` array, leaving the parent's empty so the trap reaps nothing. Do NOT backport the broken helper. See the next bullet ("Subshell-capture leak in `make_temp_file`…") for the corrected canonical pattern from `system-setup-private`'s `fix-make-temp-file-subshell-leak` branch.

- **Subshell-capture leak in `make_temp_file` — replace the helper with the inline pattern from the private repo's 2026-05-04 fix.** The canonical `make_temp_file` helper (definitions in `system-setup/utils-sys.sh`, `kubernetes/utils-k8s.sh`, `utils/services-check.sh` per the previous bullet) has a latent subshell-capture bug: every call site uses `tmp=$(make_temp_file …)` — `$()` runs the helper in a SUBSHELL, so the `TEMP_FILES+=("$tmp")` mutation lands in the subshell's array. The parent's `TEMP_FILES` stays empty; the cleanup trap iterates an empty array and reaps nothing. Confirmed in the private repo by stale `tools-update.sh.??????` files accumulating in `$HOME` after every non-success self-update path (download failure, user decline). The private `system-setup-private` repo shipped a comprehensive fix on branch `fix-make-temp-file-subshell-leak` — commits `282ca5b` (tools-update.sh), `83c0d4e` (pings-loop.sh), `e00a808` (tmux/utils-tmux.sh), `84a032f` (cross-script TTY-check normalization), `a4a26c2` (review-driven fixes). Backport as a single coordinated PR mirroring that branch's structure. The six interdependent pieces:
  - **Delete `make_temp_file`. Inline `mktemp` + `TEMP_FILES+=()` at every call site** — the parent shell then actually owns the array entry. The previous bullet's L24 recommendation to backport the helper is now superseded by this inline pattern.
  - **Adopt `~filename.tmp.XXXXXX` naming for atomic-rename temps** (leading `~`, `.tmp.` infix, mktemp's 6-char random suffix). Makes the sweep glob `~filename.tmp.??????` unambiguous — no user spontaneously creates files with this exact shape — and leftovers sort to the bottom of `ls` and read as obviously transient. Bare `$TMPDIR` temps don't need the convention. Coordinates with the atomic-rename `mktemp "${dest}.XXXXXX"` entry above (the new naming supersedes the bare `${dest}.XXXXXX` form at atomic-rename sites).
  - **Hoist `cleanup()` and `trap cleanup EXIT` to file scope (NOT inside `main()`).** Wires the trap at script load — a top-level guard that exits before `main` still reaps tracked temps. Sourcing the script now installs the trap globally; verify no caller already has its own EXIT trap that would be overwritten.
  - **Add a TTY-aware startup `sweep_stale_temps`** function that reaps any `~filename.tmp.??????` matching the new naming, with `find -mmin +10` so concurrent in-flight temps (created within the last 10 minutes by a parallel run) are spared. Defense-in-depth for SIGKILL/OOM/power-loss/exec'd-away parents that bypass the EXIT trap. Pause-and-prompt in interactive runs (with explicit `Press any key to delete and continue, Ctrl+C to abort` affordance), silent in cron/CI.
  - **TTY check for the sweep prompt MUST use `if { : </dev/tty; } 2>/dev/null; then`** (a no-op stdin redirect that probes whether `open(2)` would succeed), NOT `[[ -r /dev/tty ]]`. The bracket form only checks file permissions; under `setsid` (and similar disconnected-tty contexts) the device is world-readable but `open(2)` fails with `ENXIO`, so a subsequent `read </dev/tty` aborts under `set -e`. The bracket form is fine for `prompt_yes_no` (where the function returns immediately on the guard, so the read never happens — see the cross-script `prompt_yes_no` /dev/tty entry below), but the sweep's branch-on-TTY pattern needs the open-probe.

Defense-in-depth — pairs the trap (script exit) and sweep (subsequent invocation) with immediate per-branch cleanup.

- **Map wget exit codes to actionable error messages instead of collapsing all failures.** Most `download_script` wget branches in this repo use `wget -q ... 2>/dev/null` and treat any non-zero exit as a generic "Download failed (network/timeout)". `wget(1)` documented exit codes do distinguish: 4=network, 5=SSL, 6=auth, 8=server (HTTP 4xx/5xx), so a user with an expired credential on a wget-only host gets misdirected to look at their network. ## prompt_yes_no helpers (cross-script)

- **Add `[[ -r /dev/tty ]] || return 1` guard to every `prompt_yes_no` implementation.** Without the guard, in non-TTY contexts (cron, systemd-without-StandardInput=tty, ssh -T, CI) the `read … </dev/tty` fails under `set -e`. In an `if`-context the failure is captured by the predicate and falls through to the empty-reply branch, which returns the default. Most overwrite-and-`exec` self-update prompts in this repo default to `"y"`, so without the guard non-interactive runs auto-accept self-updates with no user consent. The fix is one line at the top of each function: `[[ -r /dev/tty ]] || return 1`. Affected sites (6 grepped 2026-04-27): `kubernetes/utils-k8s.sh:148`, `lxc/utils-lxc.sh:57`, `utils/services-check.sh:107`, `utils/rsync-two-way.sh:67`, `llm/utils-llm.sh:49`, `system-setup/utils-sys.sh:151`. ## Logging conventions (cross-script)

- **Route `print_error` through stderr (`>&2`) at every helper definition.** The Bash convention (and global agent guidance) requires errors on stderr so callers that capture stdout don't swallow them and so they appear in CI / log-pipeline error streams. Every `print_error` helper in this repo currently writes to stdout. The fix is one line per definition: `echo -e "${RED}[ ERROR   ]${NC} $1" >&2`. Affected sites (10 grepped 2026-04-27): `kubernetes/utils-k8s.sh`, `github/gh_org_delete_issues.sh`, `github/gh_org_copy.sh`, `github/gh_org_delete_repos.sh`, `lxc/utils-lxc.sh`, `utils/rsync-two-way.sh`, `llm/utils-llm.sh`, `system-setup/utils-sys.sh`, `utils/dig-all.sh`, `utils/services-check.sh`. Audit `print_warning` separately: per the private repo's review, the project's canonical helper template intentionally leaves `print_warning` on stdout (warnings are informational, not error-stream content), so DON'T blindly add `>&2` there — match the canonical template at `system-setup/utils-sys.sh` and only redirect `print_error`.

- **Audit `print_warning` calls for the `⚠ ` Unicode prefix per project convention.** The repo's helper format mandates the `⚠ ` prefix on warning messages (analogous to `✓ ` on success and `✖ ` on errors). Some `print_warning` calls in self-update / detect_download_cmd flows omit it, producing `[ WARNING ] Neither curl nor wget found...` instead of `[ WARNING ] ⚠ Neither curl nor wget found...`. Run a cross-repo audit: `grep -rn 'print_warning "[^⚠]' --include='*.sh'` and add the prefix where missing. The private repo found one such site in `pings-loop.sh:detect_download_cmd` during its 2026-04-27 review; this repo likely has more. Cosmetic-only — does not affect correctness — but tracks the project's logging convention.

## Self-update / module-update diff display (cross-script)

- **Force `--color=always` and pipe through `less -RFX` at all 16 `diff -u --color` sites.** Every `show_diff_box`-style boxed-diff in the public repo currently uses `diff -u --color` (which defaults to `--color=auto`) and writes directly to stdout with no pager. Two consequences: (1) the moment any caller pipes the output through anything other than the terminal — including `less` itself, a buffer, or a captured pipeline — `auto` resolves to "no color" and the diff renders monochrome inside an otherwise-CYAN box; (2) a self-update or module-update that touches hundreds of lines floods scrollback before the y/N prompt, pushing the prompt offscreen. The private `system-setup-private` repo's pings-loop.sh self-update plan (2026-04-27) shipped both fixes across `utils/pings-loop.sh`, `utils/tools-update.sh`, and `tmux/utils-tmux.sh`. Backport: change every `diff -u --color "$a" "$b"` to `diff -u --color=always "$a" "$b" | less -RFX` when `[[ -t 1 ]] && command -v less &>/dev/null`, falling back to bare `diff -u --color=always "$a" "$b"` otherwise. `less -R` passes ANSI through, `-F` exits if content fits one screen (so short diffs render inline with no `q` keystroke), `-X` skips alt-screen handoff so output stays in scrollback. Affected files (16 sites total, grepped 2026-04-27): `system-setup/utils-sys.sh` (2), `system-setup/system-setup.sh` (1), `kubernetes/utils-k8s.sh` (2), `kubernetes/kubernetes-setup.sh` (1), `lxc/utils-lxc.sh` (2), `lxc/_download-lxc-scripts.sh` (1), `llm/utils-llm.sh` (2), `llm/_download-ollama-scripts.sh` (1), `github/gh_org_delete_repos.sh` (1), `github/gh_org_delete_issues.sh` (1), `github/gh_org_copy.sh` (1), `utils/services-check.sh` (1). The `utils-*.sh` libraries are the natural place to extract a shared `show_diff_box` helper if pursuing DRY consolidation; otherwise apply inline at each site. Coordinate with the timeouts/`-fsSL` backport in the entry above — a single sweep across the same files covers both.

- **Sanitize non-SGR ANSI escape sequences from diff output to defend against terminal-injection.** When a self-update fetches attacker-controllable file content (the remote script could be hostile under a repo-write compromise) and renders it through `diff --color=always | less -RFX`, cursor-movement escapes embedded in the file content render directly to the user's terminal. Threats: (1) cursor-up + erase-line sequences can repaint over the diff context the user just scrolled past, hiding malicious lines; (2) more concretely, after the diff ends but before the y/N prompt prints, the file content can emit cursor escapes that draw a fake "no changes detected" line OVER the real overwrite-and-restart prompt, leveraging default-yes prompts to one-keypress acceptance. `less -R` deliberately passes ANSI through and does NOT filter cursor-movement codes; only SGR (color) is safe. The private `system-setup-private` repo's review (2026-04-27) shipped a `_sanitize_ansi` sed helper that strips two patterns: CSI sequences whose final byte is NOT `m` (cursor moves, mode toggles, erase line) and BEL-terminated OSC sequences (set-title, etc.), keeping SGR codes intact. Apply at every `show_diff_box` site (the same 16 sites listed above). Pattern: pipe diff output through `sed -E -e "s/${ESC}\\[[0-9;?]*[^0-9;?m]//g" -e "s/${ESC}\\][^${BEL}]*${BEL}//g"` where `ESC=$'\033'` and `BEL=$'\007'` (literal bytes from bash so the regex stays portable between GNU and BSD sed). Coordinate with the `--color=always` + pager backport above.
